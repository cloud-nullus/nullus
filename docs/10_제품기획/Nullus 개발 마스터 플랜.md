# Nullus v1.0 개발 마스터 플랜 (12주 + 피드백 루프)

**기간**: 2026-03-03 ~ 2026-05-25 (12주 개발) + 이후 4주 단위 피드백 루프  
**팀 규모**: Active Engineer 6명  
**릴리스 전략**: Alpha (W4) → Beta (W8) → v1 GA (W12) → 4주 피드백 루프 반복

---

## 1. 전체 구조

### 1.1 릴리스 전략 개요

```
                    개발 (12주)                          │  운영 + 확장
                                                        │
  Phase A (4주)     Phase B (4주)     Phase C (4주)      │  Cycle 1    Cycle 2    ...
 ┌──────────────┐  ┌──────────────┐  ┌──────────────┐   │ ┌────────┐ ┌────────┐
 │ 기반 + 코어  │→ │ 통합 + 확장  │→ │ 완성 + 안정화 │→  │ │ 피드백 │→│ 피드백 │→ ...
 │              │  │              │  │              │   │ │ + 개선  │ │ + 확장 │
 └──────────────┘  └──────────────┘  └──────────────┘   │ └────────┘ └────────┘
       ◆ W4              ◆ W8              ◆ W12        │    ◆ W16      ◆ W20
   Alpha 릴리스      Beta 릴리스       v1 GA 릴리스      │  v1.1       v1.2
   (내부 검증)       (공개 테스트)     (프로덕션 Ready)   │ (안정화)   (확장)
```

### 1.2 각 릴리스의 목적과 대상

| 릴리스 | 시점 | 대상 | 목적 | 품질 기준 |
|---|---|---|---|---|
| **Alpha** | W4 (3/30) | 클라우드브로 코어 멤버 5~10명 | 핵심 흐름 검증, 조기 피드백 수집 | Happy Path 동작, 버그 허용 |
| **Beta** | W8 (4/27) | 클라우드브로 전체 + 공개 모집 | 실사용 환경 테스트, 완성도 검증 | 설치 성공률 ≥85%, P0 버그 0건 |
| **v1 GA** | W12 (5/25) | 전체 사용자 (프로덕션 Ready) | 프로덕션 배포, CNCF Sandbox 준비 | 설치 성공률 ≥90%, 전 기능 완성 |

### 1.3 기능 배분 전략

PRD의 11개 기능(기능 0~10)을 3개 Phase에 걸쳐 **점진적으로 완성**합니다. 각 기능은 Alpha에서 MVP로 시작하여 Beta에서 확장, v1에서 완성됩니다.

| 기능 | PRD 공수 | Alpha (W4) | Beta (W8) | v1 GA (W12) |
|---|---|---|---|---|
| 기능 0: Organization 설정 | 2주 | ○ 단일 Admin | ○ 멤버 초대 | ● 전체 완성 |

| 기능 2: 노코드 설정 UI | 3주 | ○ 3단계 (노코드만) | ○ 5단계 + 8단계 시각화 | ● YAML 에디터 + 모드 동기화 |
| 기능 3: Golden Path 템플릿 | 1주 | ○ 1개 템플릿 | ○ 2개 템플릿 | ● 3개 템플릿 |

| 기능 5: CI/CD 템플릿 | 1주 | — | ○ 1개 템플릿 | ● 3개 템플릿 |

(○ = 해당 릴리스에서 개발, ● = 완성, — = 해당 릴리스에서 작업 없음)

### 1.4 품질 기준 및 설치 성공 정의

**설치 성공률(Installation Success Rate) 지표 정의**:
- **성공 정의**: 모든 Helm 차트 설치 완료 및 헬스체크 통과 (설치 시작 후 5분 이내)
- **실패 정의**: 설치 시작 후 30분 이내 헬스체크 미통과 또는 크리티컬 에러 발생
- **최소 표본**: 각 환경별 10회 이상 (총 30회 이상 실행 결과 합산)
- **환경 분포**: Kind(로컬) 40% / EKS 30% / GKE 30%
- **측정 기간**: 각 주요 릴리스(Alpha, Beta, GA) 전 1주간 연속 실행 데이터 기준
- **재시도 규칙**: 자동 재시도 1회까지 성공으로 인정 (수동 개입 없는 경우)

**릴리스 내용물**:

| 기능 | Alpha 수준 | 제한사항 |

| 기능 4: 자동 설치 | Deploy → 스택 설치 + 실시간 로그 | 롤백 미구현, 이력 관리 없음 |
| 기능 8: 호환성 매트릭스 | 1개 조합 검증 완료 | 경고 UI 없음 |
| 기능 10: 리소스 예상량 | 정적 계산 (CPU/Memory/Storage) | 통화 선택 없음, 비용 추정 없음 |

**Alpha 데모 시나리오**:
```
1. Nullus 웹 UI 접속 → Admin 로그인
2. K8s 클러스터 kubeconfig 등록 → 연결 상태 확인
3. Golden Path "GitLab All-in-One" 선택
4. Resources 탭에서 리소스 예상량 확인
5. "Deploy" 클릭 → 실시간 로그로 설치 과정 확인
6. 설치 완료 → 각 도구 접속 URL 확인
```

**Alpha 피드백 수집 (W4 금요일)**:
- 대상: 클라우드브로 코어 멤버 5~10명
- 방식: 라이브 핸즈온 세션 (2시간) + 구조화 설문
- 수집 항목: 설치 흐름 직관성(1~5), 설치 시간, 에러 발생 내역, 도구 조합 의견, Top 3 개선 요청

**피드백 분류**:
- **P0 (Beta Blocker)**: 설치 실패, 데이터 손실 → Phase B Week 5에서 즉시 수정
- **P1 (Beta Must)**: UX 혼란, 에러 메시지 → Phase B에서 수정
- **P2 (v1 Target)**: 추가 기능, 미세 조정 → Phase C에서 반영
- **P3 (Backlog)**: Nice-to-have → 피드백 루프에서 검토

#### 11.1 최소 운영 런북 (v1.0 GA 게이트) (M21)

1. **설치 실패 시**: 로그 분석 → 원인 파악(네트워크/권한/리소스) → `nullus rollback` 또는 재시도
2. **DB 연결 장애**: PostgreSQL 상태 확인 → 커넥션 풀 점검 → 재시작 가이드
3. **Helm 업그레이드 실패**: `helm history` 확인 → `helm rollback` 수행 → 차트 버전 정합성 체크
4. **WebSocket 끊김**: 로드밸런서 설정 확인 → 클라이언트 재연결 로직 검증
5. **인증 토큰 만료**: Keycloak 세션 설정 확인 → 토큰 갱신(Refresh Token) 흐름 점검

  - **BE-2와 풀스택 인원이 검증 스크립트 및 환경 구성을 분담하여 단일 실패지점(SPOF) 해소**

**Narwhal 레퍼런스 활용 전략**:
- DevOps W2~W3에 `known-issues.yaml` 시드 데이터 작성으로 W4 설치 성공률 향상
