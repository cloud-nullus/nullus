# Nullus 기능 고도화 및 안정화 제안서

**작성일시**: 2026-04-07
**문서 목적**: 지속적인 기능 추가(속도전)에서 한 걸음 나아가, v1 GA(프로덕션 레디) 달성을 위해 현재 구현된 시스템의 병목 현상을 파악하고 고도화 및 안정화 전략을 제안합니다.

---

## 1. 아키텍처 및 코어 로직 안정화 (Architecture Hardening)

### 1.1 인증(Auth) 및 인가(RBAC) 체계 프로덕션화
*   **현황**: 모델과 DB 스키마, 권한 미들웨어 등은 준비되어 있으나 프론트엔드-백엔드 간 OIDC 리디렉트 흐름 완전 통합, 안전한 세션(Session) 저장소 관리가 미완성 상태입니다.
*   **고도화 제안**:
    *   `gorilla/sessions` 또는 Redis 기반의 분산 세션 관리 시스템을 도입하여 API 서버의 무상태성(Stateless)을 보장해야 합니다.
    *   OAuth2/OIDC 로그인 시 프론트엔드 라우터 연동을 완료하고, 토큰 만료 및 갱신(Refresh Token) 시퀀스를 E2E 테스트로 검증해야 합니다.

### 1.2 배포 오케스트레이터(Orchestrator) 예외 처리 고도화
*   **현황**: 3-Phase DAG 기반의 정상 설치(Happy Path)와 실패 시 Full Rollback은 구현되어 있지만, 부분적인 예외나 타임아웃 처리가 미흡합니다.
*   **고도화 제안**:
    *   **Partial Success 및 Retry 로직**: 특정 Helm 릴리스만 실패했을 경우 전체를 지우지 않고 `RETRYING` 상태로 전환시킨 후 해당 Step만 재시도하는 로직이 필요합니다.
    *   **Context 취소 전파**: 배포 요청자(User)가 취소하거나 웹소켓이 끊겼을 때 리소스 누수(Goroutine Leak)를 막기 위해 철저한 `context.Context` 관리와 타임아웃 지정이 필요합니다.

### 1.3 도메인 간 이벤트 동기화 (Event-Driven Consistency)
*   **현황**: `EventBus` 추상화는 있으나 실제 비즈니스 로직(예: 클러스터 삭제 시 해당 클러스터의 배포 이력 일괄 처리)에서의 발행/구독(Pub/Sub) 결합이 약합니다.
*   **고도화 제안**:
    *   **Transactional Outbox 패턴**: 이벤트 유실 방지를 위해 DB 트랜잭션과 메세지 큐(또는 별도 Outbox 테이블)를 연동하여 시스템 통합의 원자성을 보장해야 합니다.

## 2. 관측성 및 알림 파이프라인 고도화 (Observability & Notification)

### 2.1 실제 알림 채널(Slack/Email) 배선 연결
*   **현황**: 알림 규칙(Alert Rule) CRUD 및 모니터링 조건 설정은 가능하나 실제 Webhook 발송으로 이어지지 않습니다.
*   **고도화 제안**:
    *   Notifier 인터페이스의 실제 구현체(Slack Webhook, SMTP Client) 연결을 주입(DI)하고 워커를 동작시켜야 합니다.
    *   단순 발송을 넘어, 짧은 시간 다량의 에러가 발생 시 알림 폭탄을 막기 위한 **알림 억제(Dedup/Rate Limiting)** 메커니즘을 추가해야 합니다.

### 2.2 WebSocket 스트리밍 세션 관리
*   **현황**: CI/CD 로그 및 배포 이력을 WebSocket으로 스트리밍 중입니다 (`StepTracker`).
*   **고도화 제안**:
    *   클라이언트 수 증가에 대비하여 Pub/Sub 허브의 커넥션 제한, Ping/Pong을 통한 Dead Connection 정리, 재연결(Reconnect) 시 누락된 로그의 Offset 동기화 기능이 추가되어야 합니다.

## 3. 프론트엔드 UX 안정화 (Frontend UX Stability)

### 3.1 폼(Form) 상태 보존 및 에러 인터페이스
*   **고도화 제안**:
    *   5단계~6단계 위저드 진행 중 실수로 새로고침을 하거나 이전 버튼을 누를 경우를 대비하여 `localStorage`를 활용한 임시 상태 저장이 필요합니다.
    *   에러 발생 시 일반적인 토스트 알림 외에도 트러블슈팅 가이드나 명확한 힌트(로그 확인 버튼 등)를 제공하는 **Actionable Error Feedback** UI가 요구됩니다.

## 4. 테스트 인프라 및 QA 고도화

### 4.1 동시성/부하 테스트 (Load Testing)
*   **현황**: 높은 비율의 기능/통합 테스트가 작성되어 있으나, 다중 사용자의 동시 접근 상황에 대한 검증은 보강이 필요합니다.
*   **고도화 제안**:
    *   동시에 여러 개의 DevSecOps 스택을 각기 다른 클러스터에 배포하는 시나리오를 k6나 Locust를 활용하여 부하 테스트. DB 커넥션 풀과 고루틴 스케줄러의 안정성 점검.

### 4.2 인프라 의존적 Edge Case 테스트
*   **고도화 제안**:
    *   클러스터 인증 만료, 네트워크 파티션(통신 단절) 등 인프라 레벨의 장애 상황(Chaos Testing)을 고의로 주입하여, Nullus가 오작동 없이 안전하게 `FAILED` 처리 및 권고 사항을 리포트하는지 점검해야 합니다.

## 💡 결론 및 우선순위 제언

현재의 개발 모멘텀을 유지하면서도 GA 수준의 품질을 달성하려면, 신규 기능(예: CLI, 수출/가져오기 등 Phase 2 항목)의 착수를 뒤로 미루고 다음 과제에 팀 리소스를 재분배할 것을 권장합니다.

1. **최우선 (P0)**: 미결된 인증/세션 체계의 프로덕션 반영 및 Alert Notification (Slack 등) 흐름 완성
2. **우선 (P1)**: 배포 실패 시 `예외 복구(Retry) 매커니즘` 및 WebSocket 예외 처리 고도화
3. **다음 (P2)**: 부하 테스트 및 다국어/접근성 최종 튜닝
