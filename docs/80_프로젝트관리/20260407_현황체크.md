### ⚠️ 3. 잠재적 리스크 및 기술 부채 (GA 릴리스를 위한 보완점)
개발 속도가 빠른 반면, 설계 문서 대비 아직 적용되지 않은 몇몇 **아키텍처 및 미구현 항목**들이 남아있습니다. 이들은 Beta(W8) 보다는 **v1 GA(W12) 릴리스 전까지 안정화 기간(W9~W11)**에 채워 넣어야 할 기술 과제들입니다. (`docs/20_아키텍처/Nullus_설계_대비_미구현_항목.md` 참조)

1.  **인증 / 세션 아키텍처의 프로덕션 레디:**
    *   현재 프론트엔드의 완전한 OIDC 리디렉트 흐름 코드가 아직 Placeholder 상태이며, 백엔드의 `User API` 및 DB 기반의 `Session/RBAC Policies` 연결이 약식(Mock 헤더 읽기)으로 처리되어 있어 W9의 인증 전환 기간에 추가 작업이 필요합니다.
2.  **클러스터 내부 Operator / 알림 파이프라인 부재:**
    *   기획 시 런타임 클러스터 내부에 Operator/Agent가 모니터링을 관장하기로 했으나, 현재는 중앙 서버 발송 방식(Helm 등)만 사용합니다.
    *   Alert Rule CRUD는 생성되었지만, 조건 충족 시 **실제 Slack이나 Email로 발송**하는 Notification 모듈 플러그링이 아직 되어있지 않습니다.
3.  **예외 상황(Edge Case) 대응 로직:**
    *   설치 실패 후 `RETRYING` 상태나 `PARTIAL_SUCCESS` 등 세분화된 배포 상태 머신 확장이 필요하며, API 레이어에서 `설치 재시도(Retry)` 엔드포인트 마련이 필요합니다.

### 🎯 총평 및 다음 스텝 제언
현재 기획된 요구사항이나 UI, 핵심 서비스 연동(Helm) 기능 구현의 진척 속도가 탁월합니다. 
진행 속도에 여유가 있으므로 남은 **Sprint 3B (W7)** 와 **Beta 안정화 주간 (W8)** 기간에는 기능 추가보다 **실사용자 QA**, E2E 테스트(Playwright) 보완, 그리고 위에 열거된 **알림 발송 연결 및 Edge Case(배포 실패 시 자동 롤백/재시도)의 완성도를 높이는 것**에 집중하면 매우 안정적인 수준의 4월 말 Beta 릴리스가 가능할 것으로 진단됩니다.
