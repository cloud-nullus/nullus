# Nullus GA 조기 달성 및 고도화 집중 스프린트 계획

**목표**: 기존 5월 말 예정이었던 GA 출시를 한 달 앞당겨 **4월 27일**에 달성하고, 남은 3주간 프로덕션 수준의 안정화 및 고도화를 완성합니다.
**형태**: 사용자 페르소나별 기능 단위 및 Use Case 중심의 액션 플랜

---

## 1. 계획 요약 및 기존 계획 대비 누락된 핵심 보완(Missing Parts)

기존 계획이 "기능의 빠른 동작"에 집중했다면, 프로덕션 환경의 v1.0 에서는 **"안정적인 실패 처리"**와 **"시스템 밖(외부)과의 연결"**이 필요합니다. 이번 스프린트에서는 다음 누락 요소들을 중점적으로 보완합니다.

1. **상태 머신 및 장애 복구 (Orchestrator Stability)**: 망 분리나 노드 다운으로 인한 배포 중단 시, 처음부터 재시작하지 않고 끊긴 Step(Phase)부터 이어하는 **이어하기(Resume)** 와 **선택적 재시도(Partial Retry)** 기능.
2. **이벤트와 외부 시스템 연동 (External Wiring)**: 알림 규칙만 등록하고 끝나던 것을 넘어, 실제 Slack Webhook / Email SMTP 서버와 실질적인 발송(Notifier) 연결 배선작업.
3. **감사 로그 (Audit Log)**: "누가 언제 비용과 권한에 영향을 미치는 스택을 수정했는가?"에 대한 추적성 보완 (Admin 페르소나 필수 요소).
4. **프론트엔드 연결 복원 (UI Resiliency)**: WebSocket 기반의 로그 스트리밍 중 네트워크 끊김 발생 시 `offset` 기반의 재연결 및 누락 로그 자동 Fetch 지원.

## 2. 사용자 페르소나별 고도화 및 v1.0 완성 마일스톤

### 2.1 Admin (플랫폼 관리자)

Admin은 Nullus 시스템이 조직의 보안 및 인프라 정책 아래 안전하게 통제되도록 보장합니다.

| 기능 단위 | Use Case | 완료 기준 (DoD) & 고도화 포인트 |
| :--- | :--- | :--- |
| **인증/세션 고도화** | UC-A1. "사용자는 Keycloak 기반의 기업 SSO를 통해 1회 로그인으로 Nullus에 안전하게 접근한다." | - 현재 Mock 헤더 리딩 기반에서 완전한 React OIDC 리디렉션 통합으로 변경<br>- 백엔드 OIDC Token 검증 미들웨어 완비<br>- 분산 환경 대응을 위한 세션 클러스터링(또는 Redis 도입) |
| **권한 통제 (RBAC)**| UC-A2. "관리자는 팀원에게 데브옵스 권한만 부여하고, 조직 설정 및 타 클러스터 삭제를 방지한다." | - 미구현되었던 `/api/v1/users` 상위 API 완성<br>- Role별 라우팅 UI 가시성 및 API 권한 인가 완전 분리 테스트 통과 |
| **감사 로그 (신규)** | UC-A3. "관리자는 보안 사고 발생 시, 누가 특정 스택의 클러스터 정보를 언제 변경했는지 추적한다." | - (*놓친 부분 보완*) 스택의 C/U/D 및 클러스터 변경 내역을 6개월 이상 보존하는 `audit_logs` 테이블 구축 및 조회 UI 제공 |
| **보안 강화** | UC-A4. "관리자는 등록된 타겟 클러스터의 민감한 Kubeconfig가 탈취되지 않게 관리한다." | - 업로드 시 AES-256-GCM 암호화 로직 재검토 및 Key Rotation 기능 확보 |

### 2.2 DevOps Engineer (플랫폼/데브옵스 엔지니어)

DevOps Engineer가 겪게 될 "불안정한 쿠버네티스 인프라 환경"에서의 견고함을 확보하는 것이 가장 중요합니다.

| 기능 단위 | Use Case | 완료 기준 (DoD) & 고도화 포인트 |
| :--- | :--- | :--- |
| **배포 장애 복구** | UC-D1. "설치 도중 타임아웃 발생 시, 스택 전체가 삭제되지 않고 실패한 컴포넌트부터 재시도(Retry)한다." | - (*놓친 부분 보완*) `RETRYING`, `PARTIAL_SUCCESS` 상태 전이 도입<br>- `POST /stacks/{id}/deploy/retry` 엔드포인트 마련 및 타임아웃 복구 테스트 |
| **상태 및 로그 추적** | UC-D2. "서버가 재시작되어도 과거의 설치 로그와 현재 진행 중인 배포 단계를 잃지 않고 연결한다." | - DB 영속화된 배포 로그(`deployment_logs`)의 Pagination 조회 API 완비<br>- 서버 재기동 시 In-memory 큐에 남은 진행 상태를 DB와 싱크하는 복원 로직 |
| **알림 외부 연동** | UC-D3. "설치가 완료되거나 파이프라인에서 Error 이벤트가 발생하면 팀 Slack 방으로 노티를 받는다." | - (*놓친 부분 보완*) Notification Worker 프로세 가동<br>- 중복 알림 억제(Dedup) 기능을 가진 알림 발송 채널 구현 (Slack 배선) |
| **Helm Edge-Case** | UC-D4. "크기 초과 에러나 권한 에러 등 엣지 케이스들을 알아서 우회하여 설치 성공률 90%를 넘긴다." | - `known-issues` 패턴을 10개 이상 확보하여 API에서 자동 회피 (예: CRD Server-side apply 플래그 자동 전환) |
| **동시 배포 안정성** | UC-D5. "동시에 5개의 스택을 배포해도 Nullus 플랫폼 자체가 뻗지 않는다." | - Helm 실행 Go routine 풀 캐파 관리 (Rate Limit 적용)<br>- 부하에 따른 Kube-API 서버 Rate limit 에러 대응 백오프(Backoff) 로직 구현 |

### 2.3 Developer (개발자)

개발자는 UI/UX 상의 끊김 없고 부드러운 앱 배포 경험(Self-Service)을 얻어야 합니다.

| 기능 단위 | Use 시나리오 | 완료 기준 (DoD) & 고도화 포인트 |
| :--- | :--- | :--- |
| **앱 배포 UI 안정화** | UC-V1. "앱 환경변수를 10개 이상 복잡하게 넣고 배포 버튼을 누르다 취소해도 창이 프리징되지 않는다." | - 프론트엔드의 비동기 배포 요청 간 컴포넌트 상태 보존(`localStorage` 등 활용)<br>- 앱 배포 매니페스트 프리뷰 로딩 속도 최적화 |
| **로그 재연결 (UI)** | UC-V2. "노트북 뚜껑을 덮었다 열어도, 배포 중이던 CI/CD 파이프라인 진행률과 로그가 다시 정상적으로 동기화된다." | - (*놓친 부분 보완*) WebSocket 세션 끊김 시, `offset`을 기억하고 Reconnect하여 하단 스트리밍을 이어가는 재연결 로직 구현 |
| **알림 피드백** | UC-V3. "개발자가 Git Push 후 Slack을 통해 내 파이프라인 배포 성공 여부를 확인한다." | - Developer가 배포한 CI/CD 작업과 DevOps가 설정한 Alert Rule 간 연결. (Dev/Ops 결합 알림) |

## 3. 주차별 역산 마일스톤 (4월 8일 ~ 4월 27일)

총 3주간의 전력 질주를 위해 스쿼드의 집중도를 아래와 같이 재편성합니다. 기능 추가보다는 **테스트, 수정, 외부 연동, 복구**에 80% 리소스를 투입합니다.

### Week 1 (4/8 ~ 4/14): 고도화 인프라 결합 주간 (통합의 빈틈 메우기)
* **백엔드**: OIDC 실결합 및 세션 스토리지 아키텍처 완성 확정. / 알림 Notifier Worker(Slack 등) 배선 구축 및 테스트.
* **프론트엔드**: WebSocket 재접속(Reconnection) 로직, OIDC 리디렉션 구현, 에러 발생 시의 폴백 UI (Actionable Error) 디자인 및 반영.
* **인프라**: Kubeconfig/Helm Client 동시성 관리 래퍼(Rate limiter) 적용.

### Week 2 (4/15 ~ 4/21): Edge-Case 방어 및 인젝션 주간 (단단함 확보)
* **오케스트레이터**: 배포 실패 시 `Retry` 액션 개발 및 망연결 단절, Kube API 429 Too Many Requests, Helm 타임아웃 강제 주입(Chaos Testing). 부분 롤백 로직 테스트.
* **DevOps Use-Cases**: 통합 E2E 테스트(Playwright) 강화 - Golden Path 선택부터 모니터링 연동까지 Happy Path 100% 지속 통과 확인.
* **추적/보안**: 감사 로그(Audit) 이벤트 버스 발행 및 DB 기록 안착 확인.

### Week 3 (4/22 ~ 4/27): v1.0 GA 컷오버 및 버그 바운티
* **품질 확인**: "설치 성공률 90% 이상" 및 "P0(치명적) 버그 0건" 목표를 실제 EKS, GKE, Kind 3가지 환경 배포를 통해 확인.
* **UI/UX 폴리싱**: 영/한 다국어 텍스트 정리, 로딩 스피너 및 프로그레스 툴팁 일관성 적용.
* **릴리스 패키징**: `V1.0 GA` 릴리스 태깅 및 사용 설명서(Release Notes, Quick Start 안내서) 완성.
* **4/27 (월)**: v1.0 GA 공식 릴리스.
