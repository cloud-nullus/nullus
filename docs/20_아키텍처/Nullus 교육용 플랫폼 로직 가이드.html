Nullus 전체 로직을 함수 기준으로 읽는 페이지

이 페이지는 추상 아키텍처 그림이 아니라, 실제 코드가 어떤 순서로 호출되고 어떤 알고리즘으로 동작하는지 이해하는 데 초점을 둡니다. 각 섹션은 진입점, 핵심 구조체, 처리 알고리즘, 봐야 할 파일 순서로 정리했습니다.

현재 코드 기준 함수 흐름 중심 교육용 설명 HTML 단일 페이지

이 페이지를 읽는 법

1. Handler

HTTP 요청을 받아 입력을 파싱하고, 어떤 usecase를 호출할지 결정합니다.

2. UseCase

실제 업무 절차를 조립합니다. 상태 전이, 저장, 비동기 시작, 실패 처리의 중심입니다.

3. Port

DB나 Helm 같은 바깥 세상에 대해 "무슨 기능이 필요하다"만 인터페이스로 정의합니다.

4. Adapter

Port를 실제로 구현합니다. PostgreSQL, Prometheus, Helm, middleware가 여기에 해당합니다.

중요한 해석 포인트 현재 Nullus는 "각 모듈이 완전히 독립 실행되는 분산 시스템"이 아니라, cmd/api/main.go가 모든 모듈을 한 프로세스 안에서 조립하는 모듈러 모놀리스입니다.

서버 기동과 배선

프로젝트를 이해할 때 가장 먼저 봐야 하는 파일은 cmd/api/main.go입니다. 이 파일은 "무엇을 만들고, 무엇을 어디에 연결하는지"를 전부 보여줍니다.

핵심 배선 순서

1. 설정과 DB 연결config.LoadConfig()로 설정을 읽고, pgxpool.NewWithConfig()로 PostgreSQL 풀을 만듭니다.
2. 모듈별 repository 생성예: NewPostgresStackRepository, NewPostgresPipelineRepository, NewPostgresAlertRuleRepository
3. usecase 생성repository와 외부 포트를 usecase에 주입합니다. 예: NewInstallStack, NewDeployPipeline, NewGetDashboard
4. handler 생성각 handler는 필요한 usecase를 받아 REST API 진입점이 됩니다.
5. 인증 모드 분기개발 모드에서는 auth를 끄고, 운영 모드에서는 DualAuthMiddleware로 세션/JWT를 묶습니다.
6. 라우트 등록RegisterRoutes() 호출로 각 모듈 엔드포인트를 Echo에 연결합니다.

개발 모드와 운영 모드 차이

if cfg.Server.Mode == "development" {
  // auth 미들웨어 비활성
  admin = v1.Group("/admin")
  stacks = v1.Group("/stacks")
} else {
  sessionMW := AuthMiddleware()
  oidcMW := JWTAuthMiddleware(...)
  authMW := DualAuthMiddleware(cfg.Auth.Mode, sessionMW, oidcMW)

  admin = v1.Group("/admin", authMW, RequireRole("admin"))
  stacks = v1.Group("/stacks", authMW, RequireRole("admin", "devops"))
  cicd = v1.Group("/cicd", authMW, RequireRole("admin", "devops", "developer"))
}
가장 먼저 볼 파일cmd/api/main.go - 전체 시스템의 DI 컨테이너 역할
왜 중요한가각 모듈이 어떤 외부 구현체를 쓰는지, 어떤 인증 규칙으로 묶였는지, 어떤 라우트가 열려 있는지 한 번에 볼 수 있습니다.

Stack 설치 알고리즘

이 프로젝트에서 가장 복잡한 핵심 로직입니다. 설치를 "한 번에 끝내는 함수"가 아니라, 상태 전이 + DAG 기반 step 실행 + 실시간 로그 스트리밍 조합으로 구현돼 있습니다.

진입점

HTTP

POST /api/v1/stacks/:id/deploy

파일: internal/stack/adapter/handler/deploy_handler.go

UseCase

InstallStack.Execute()

파일: internal/stack/usecase/install_stack.go

실행체

Orchestrator.ExecuteStep() → HelmInstaller.Install()

파일: adapter/helm/orchestrator.go, installer.go

핵심 구조체

InstallStack

  • stackRepo: 상태 저장
  • streamer: 실시간 로그 fan-out
  • executor / dynamicExecutorFunc: step 실행체
  • kubeconfigProvider: 대상 클러스터 kubeconfig 조회
  • tokenRegistry: OpenBao 기반 토큰 소스 등록

installDAG

  • installing_cert_manager
  • installing_minio, installing_openbao
  • installing_gitlab, installing_argocd, installing_runner
  • installing_prometheus, installing_grafana, integration_check

DAG가 뭐냐

DAG는 Directed Acyclic Graph의 약자입니다. 한국어로 풀면 방향이 있고, 순환이 없는 작업 연결도 정도로 보면 됩니다.

쉽게 말하면

"이 작업은 저 작업이 끝난 뒤에만 할 수 있다"는 규칙을 그래프로 표현한 것입니다. 즉, 작업 순서를 코드에 하드코딩한 긴 if문 대신, 선행 조건으로 관리하는 방식입니다.

왜 쓰나

설치 단계가 많아질수록 순서 의존성이 복잡해집니다. DAG를 쓰면 어떤 단계가 무엇을 기다리는지 명확해지고, 일부 단계를 끄거나 재개할 때도 흐름을 관리하기 쉬워집니다.

예시:
cert-manager ----┐
                 ├--> postgresql --> minio --> object-storage-secret
metrics-server --┘

openbao 는 postgresql, minio 뒤에 실행
gitlab 은 openbao, object-storage-secret 뒤에 실행
runner 는 openbao, gitlab 뒤에 실행
Nullus에서 DAG를 어떻게 읽으면 되나 installing_gitlab step에 installing_openbao, installing_object_storage_secret 같은 dependency가 걸려 있으면, GitLab 설치는 그 앞 단계들이 모두 끝날 때까지 시작되지 않습니다. 즉, 앞선 준비물이 다 갖춰져야 다음 단계가 열린다고 이해하면 됩니다.

설치 알고리즘

DeployHandler.Deploy()
  -> compatibility gate 확인
  -> InstallStack.Execute(StackID)

InstallStack.Execute()
  1. stackRepo.GetByID(StackID)
  2. resolveExecutor()로 kubeconfig 기반 실행체 준비
  3. Continue 모드면 ResumeFromStep 계산
  4. 상태를 validating 으로 전이하고 저장
  5. goroutine 으로 run() 시작

InstallStack.run()
  1. 로그 히스토리 초기화 여부 결정
  2. validate step 로그 기록
  3. 상태를 installing 으로 전이
  4. runPhases() 실행
  5. 상태를 configuring 으로 전이
  6. 상태를 health_check 로 전이
  7. verifyDeployment() 실행
  8. 성공 시 completed 로 전이
  9. OpenBao 사용 시 token source 등록

InstallStack.runPhases()
  while processed step 수 < 전체 DAG step 수:
    for each step in installDAG:
      - 이미 처리했으면 skip
      - dependency 미완료면 skip
      - resume 모드인데 아직 resume step 전이면 completed 처리 후 skip
      - stack 이 취소/삭제/실패 상태인지 ensureDeploymentActive()
      - stepEnabledChecker 가 false 면 skip 로그만 기록
      - markStepStarted()
      - StartStepRuntimeTail() 로 런타임 로그 tail 시작
      - executeStep()
      - cert-manager 실패면 executor 재생성 후 1회 재시도
      - openbao step 완료 후 runOpenBaoHealthGate()
      - StepRuntimeLogs() 수집
      - markStepCompleted()
    if 한 바퀴 돌았는데 progressed == false:
      DAG blocked 에러 반환

중요한 구현 디테일

실시간 로그

MemoryStreamer가 deploymentID 별 history와 subscriber 채널을 들고 있습니다. StreamLogs()는 WebSocket을 upgrade 한 뒤 이 채널을 구독합니다.

실패 처리

현재 코드는 실패 시 자동 롤백보다 failed 상태로 멈추고 Continue를 유도하는 방향입니다. handleFailure()는 failed 전이와 안내 로그를 남깁니다.

선택적 step

도구 선택에 따라 step이 꺼질 수 있습니다. 이때 executor 쪽에서 skip을 반영하고, usecase는 processed/completed 상태를 맞춰 DAG가 막히지 않게 합니다.

Helm chart를 실제로 어떻게 읽고 설치하나

여기서 중요한 점은 모든 step이 Helm chart 설치는 아니라는 것입니다. Stack 설치 엔진은 step마다 아래 셋 중 하나를 고릅니다.

1. Helm chart 설치

cert-manager, postgresql, gitlab, argocd, grafana 같은 도구는 Helm chart로 설치합니다.

2. Raw YAML apply

installing_openbao는 현재 chart가 아니라 코드로 만든 manifest를 그대로 apply합니다.

3. 특수 함수 실행

object_storage_buckets, database_connection_check처럼 연결/검증용 step은 전용 함수가 수행합니다.

어떤 chart를 어디서 받는가

step별 Helm chart 정보는 stack_helm_step_configs 테이블에서 조회합니다. 오케스트레이터는 이 테이블을 읽어 release / chart / repo / version을 조합합니다.

예시 chart source
installing_cert_manager  -> cert-manager / https://charts.jetstack.io
installing_postgresql    -> postgresql / https://charts.bitnami.com/bitnami
installing_gitlab        -> gitlab / https://charts.gitlab.io/
installing_argocd        -> argo-cd / https://argoproj.github.io/argo-helm
installing_prometheus    -> kube-prometheus-stack / https://prometheus-community.github.io/helm-charts
installing_gateway       -> oci://docker.io/envoyproxy/gateway-helm

버전은 어디서 결정되나

현재 실제 기준

설치에 쓰는 Helm chart 버전은 현재 latest가 아니라, stack_helm_step_configs.version에 저장된 값입니다. 예를 들어 GitLab은 8.7.2, Argo CD는 7.7.16처럼 DB에서 조회됩니다.

템플릿과의 관계

domain.Template에는 HelmVersion, AppVersion 필드가 있지만, 현재 설치 로직은 그 값을 직접 읽어 chart version으로 쓰지 않습니다.

현재 구조
Template.Tools[*].HelmVersion / AppVersion
  -> 카탈로그/설명/호환성 표현 용도

stack_helm_step_configs.version
  -> 실제 Helm install 에 사용되는 버전
PM이 이해해야 할 차이 지금 Nullus는 "템플릿에 적힌 버전으로 바로 설치"하는 구조가 아닙니다. 실제 설치 버전은 stack_helm_step_configs 테이블이 기준입니다. 즉, 템플릿 화면에 보이는 버전과 실제 설치 chart 버전이 항상 1:1로 연결된다고 가정하면 안 됩니다.

다만 일부 step은 설정에 따라 chart 종류 자체가 바뀌는 예외가 있습니다.

  • installing_log_search: opensearch 또는 elasticsearch
  • installing_opentelemetry: tempo, jaeger, 기본 opentelemetry-collector

이 경우에도 최종 chart/버전 결정은 여전히 resolveChartSpecForStep() 안의 코드가 기준입니다.

Helm 설치 알고리즘

runPhases()
  -> executeStep()
  -> Orchestrator.ExecuteStep(step)

Orchestrator.ExecuteStep(step)
   1. DB에서 step의 ChartSpec 조회
   2. step 이 disabled 면 skip
   3. manifest step 이면 YAML apply
   4. chart step 이면:
      - resolveChartSpecForStep() 로 최종 chart 결정
      - valuesForStep() 로 values 조립
      - installer.Install() 호출

HelmInstaller.Install(req)
  1. newActionConfig() 로 Helm SDK 실행 컨텍스트 생성
  2. action.NewInstall(cfg)
  3. RepoURL / Version / Namespace / ReleaseName 세팅
  4. LocateChart() 로 외부 repo 또는 OCI에서 chart 위치 탐색
  5. loader.Load() 로 chart 로드
  6. RunWithContext(chart, values) 로 실제 install
  7. 같은 release 가 이미 있으면 upgrade 또는 reinstall 시도

values는 어떻게 만들어지나

internal/stack/adapter/helm/helm-values.go에서 차트 values를 단계적으로 합칩니다.

기본값 + 리소스 기본값

step별 기본 values를 먼저 가져오고, CPU/메모리/스토리지 같은 resource default를 merge합니다.

Stack 설정 반영

예: GitLab은 앞에서 설치한 PostgreSQL/MinIO를 사용하도록 host, secret, object storage 연결값을 주입합니다.

YAML override 반영

사용자가 step별 override를 넣으면 마지막에 덮어씁니다. 그래서 기본 chart라도 최종 values는 stack마다 달라질 수 있습니다.

동적 chart 선택

installing_log_search는 OpenSearch/Elasticsearch 중 선택되고, installing_opentelemetry도 Tempo/Jaeger 등으로 바뀔 수 있습니다.

스택 OSS 리소스를 실제 설치에 반영하는 메커니즘

리소스 관련 로직은 두 층으로 나뉩니다.

1. 사전 추정(Estimate)

사용자가 스택 구성을 입력하면 /estimate API가 대략적인 CPU/메모리/스토리지 요구량을 계산합니다. 이건 "설치 전 계산기"에 가깝습니다.

2. 실제 설치 반영(Helm values)

설치 시점에는 DB에 저장된 resource-defaults를 읽어서 Helm values의 resources.requests/limits에 주입합니다. 이게 실제 배포 spec에 들어갑니다.

사전 추정 로직

POST /api/v1/stacks/estimate
  -> ResourceHandler.Estimate()
  -> StackConfig 에서 enabled tool 목록 추출
  -> CalculateResources.Execute()

CalculateResources.Execute()
  1. toolBaselineMap 에서 툴별 baseline 자원 조회
  2. developers / build frequency 기반 scale factor 계산
  3. tool별 CPU/Memory/Storage 계산
  4. weekly commits 기반 artifact storage 추가
  5. 월 비용 추정까지 계산

예를 들어 baseline은 이런 식으로 정의돼 있습니다.

gitlab-ce       -> CPU 4 / Memory 8Gi / Storage 30Gi
gitlab-runner   -> CPU 2 / Memory 4Gi / Storage 10Gi
argocd          -> CPU 2 / Memory 3Gi / Storage 5Gi
prometheus      -> CPU 1 / Memory 4Gi / Storage 20Gi
grafana         -> CPU 1 / Memory 2Gi / Storage 5Gi
중요 이 추정값은 "상담용/가이드용 계산"입니다. 실제 Kubernetes Deployment/StatefulSet spec에 바로 쓰이는 값은 아닙니다.

실제 설치 반영 로직

Orchestrator.valuesForStep(step, spec)
  1. step 기본 values 복사
  2. resourceDefaultValuesForStep(step, cfg) merge
  3. namespace/shared service values merge
  4. YAML override 있으면 마지막에 덮어쓰기

핵심은 resourceDefaultValuesForStep() 입니다.

resourceDefaultValuesForStep(step)
  1. resourceDefaultKeyForStep(step) 로 tool key 결정
     - installing_gitlab     -> gitlab-ce
     - installing_argocd     -> argocd
     - installing_prometheus -> prometheus
     - installing_grafana    -> grafana
     - installing_minio      -> minio
  2. loadResourceDefault(key) 로 DB resource default 로드
  3. toK8sResourceValues() 로 requests/limits 형태로 변환
  4. step별 chart 구조에 맞게 values tree 구성

왜 step마다 방식이 다르나

단순 차트

MinIO, Grafana, Runner 같은 건 거의 resources 한 덩어리만 넣으면 됩니다.

복합 차트

GitLab, ArgoCD, Prometheus는 하위 컴포넌트가 많아서 webservice, sidekiq, repoServer, alertmanager 등 여러 위치에 나눠 넣어야 합니다.

공급자별 분기

OpenSearch/Elasticsearch, Tempo/Jaeger처럼 실제 차트 종류가 바뀌면 resource key와 values 구조도 함께 달라집니다.

GitLab 리소스 반영 예시

installing_gitlab
  -> resource key: gitlab-ce
  -> DB의 gitlab-ce 기본 자원값 로드
  -> 컴포넌트별로 분배:
     webservice      -> scaled(0.12) + 최소/최대 clamp
     sidekiq         -> scaled(0.10) + 최소/최대 clamp
     toolbox         -> scaled(0.05)
     gitaly          -> scaled(0.20)
     kas             -> scaled(0.12)
     gitlab-exporter -> scaled(0.05)
     registry        -> scaled(0.12)
     redis master    -> 별도 redisMasterScaled()
     prometheus      -> scaled(0.08)

즉 GitLab은 "gitlab-ce 1개의 리소스 값"을 그대로 넣는 게 아니라, 그 값을 여러 내부 컴포넌트 비율로 쪼개서 적용합니다.

Prometheus 리소스 반영 예시

installing_prometheus
  -> resource key: prometheus
  -> 같은 resource 기본값을
     prometheus.prometheusSpec.resources
     alertmanager.alertmanagerSpec.resources
     kube-state-metrics.resources
     prometheusOperator.resources
     prometheus-node-exporter.resources
  에 주입

리소스 기본값은 어디서 오나

저장 구조

domain.ResourceDefault는 ToolKey, CPU request/limit, memory request/limit, storage request/limit을 가집니다.

조회 방식

loadResourceDefault()가 repository에서 한 번 목록을 읽어 메모리에 캐시하고, step별 tool key로 꺼내 씁니다.

ResourceDefault 예시 필드
ToolKey
CPURequest / CPULimit
MemoryRequestGi / MemoryLimitGi
StorageRequestGi / StorageLimitGi
PM 관점 요약 현재 Nullus는 "스택 규모를 먼저 추정"하는 기능과, "실제 설치 시 Helm values에 리소스를 주입"하는 기능이 분리돼 있습니다. 전자는 계획/상담용이고, 후자는 실제 배포 spec 생성용입니다.

cannot re-use a name that is still in use가 나오면 실패로 끝내지 않고 상태를 보고 upgrade 또는 reinstall를 시도합니다.

Airgap 모드

NULLUS_HELM_OCI_REGISTRY가 있으면 일반 Helm repo 대신 내부 OCI registry에서 chart를 읽습니다.

Gateway 특수 처리

Gateway는 chart 설치 후 끝나지 않고 CRD 보장, TLS 보정, route manifest apply까지 이어집니다.

에어갭 설치 로직

에어갭 모드에서는 외부 Helm repository에서 chart를 내려받지 않고, 미리 준비된 내부 OCI registry에서 chart를 읽습니다. 분기 기준은 환경변수 NULLUS_HELM_OCI_REGISTRY 입니다.

HelmInstaller.Install(req)
  if NULLUS_HELM_OCI_REGISTRY != "":
    -> installAirgapOCI(req)
  else:
    -> 일반 Helm repo install 경로

에어갭 경로에서 실제로 하는 일

installAirgapOCI(req)
  1. registry = NULLUS_HELM_OCI_REGISTRY
  2. chartBaseName(req.ChartName) 으로 chart 이름의 마지막 segment 추출
     예: cert-manager -> cert-manager
     예: oci://docker.io/envoyproxy/gateway-helm -> gateway-helm
  3. ociRef = oci://{registry}/{baseName}
  4. values 가 있으면 임시 yaml 파일 생성
  5. installOCIChartWithHelmCLI(...) 호출
  6. 설치 후 가능하면 release status 조회
  7. 조회 실패해도 deployed 로 간주한 synthetic result 반환

일반 모드와 차이

일반 모드는 LocateChart()로 외부 repo URL을 찾아가지만, 에어갭 모드는 repo URL을 거의 무시하고 내부 OCI 참조값으로 재조합합니다.

왜 Helm SDK가 아니라 CLI fallback을 쓰나

에어갭/OCI 환경에서는 registry 접근 조건이 까다로울 수 있어서, 현재 구현은 installOCIChartWithHelmCLI()를 통해 CLI 기반 설치를 사용합니다.

예시
환경변수:
  NULLUS_HELM_OCI_REGISTRY=kind-registry:5000/charts

요청 chart:
  postgresql

실제 읽는 OCI ref:
  oci://kind-registry:5000/charts/postgresql

에어갭 설치 준비물

1. 내부 OCI registry

클러스터에서 접근 가능한 내부 registry가 필요합니다. 예: kind-registry:5000/charts

2. chart 사전 적재

인터넷 없이 설치하려면 필요한 Helm chart를 미리 OCI 형태로 내부 registry에 push 해둬야 합니다.

3. 이미지 사전 적재

chart만 있어도 부족합니다. chart가 참조하는 컨테이너 이미지도 별도 미러링돼 있어야 실제 pod가 뜹니다.

내부 registry에 미리 있어야 하는 chart 예시

cert-manager
metrics-server
postgresql
minio
gitlab
argo-cd
gitlab-runner
kube-prometheus-stack
grafana
loki
opensearch 또는 elasticsearch
opentelemetry-collector 또는 tempo 또는 jaeger
gateway-helm

실제로 필요한 목록은 사용자가 켠 기능에 따라 달라집니다.

  • OpenBao를 쓰면 installing_openbao는 chart가 아니라 manifest라 chart 적재 대상이 아닙니다.
  • Logging search를 OpenSearch로 쓰면 opensearch chart가 필요합니다.
  • Trace layer를 Jaeger로 바꾸면 jaeger chart가 필요합니다.

PM이 이해해야 할 운영 포인트

에어갭 설치 성공 조건
  = 내부 registry 에 chart 존재
  + 내부 registry/이미지 저장소 에 컨테이너 이미지 존재
  + 클러스터가 그 registry 에 접근 가능
  + NULLUS_HELM_OCI_REGISTRY 가 올바르게 설정됨

왜 chart만으로 끝나지 않나

Helm chart는 Kubernetes 리소스 설계도일 뿐입니다. 실제 Pod는 chart가 지정한 이미지 태그를 pull 해야 하므로 이미지 미러링이 따로 필요합니다.

실패가 나는 대표 경우

chart는 내부 registry에 있는데 이미지가 외부 registry를 가리키면 install은 성공해도 Pod는 ImagePullBackOff로 실패할 수 있습니다.

PM 관점 요약 에어갭 설치는 "인터넷이 없는 환경에서도 설치할 수 있게, 외부 Helm repo 대신 내부 registry에 미리 적재한 chart를 사용"하는 기능입니다. 즉 제품 동작은 비슷하지만, chart를 읽어오는 소스만 인터넷 repo에서 내부 OCI registry로 바뀐다고 보면 됩니다.
PM 관점 요약 Stack 설치는 "Helm chart 여러 개를 순서대로 설치하는 기능"이라기보다, Helm chart 설치 + YAML apply + 연결값 주입 + 후처리 검증을 묶은 설치 엔진입니다. 그래서 같은 "GitLab 설치"라도 단순 install이 아니라 PostgreSQL/MinIO/OpenBao와의 연결 관계까지 함께 고려됩니다.

OpenBao가 OSS 연결에 쓰이는 방식

OpenBao는 여기서 "스택 안에 포함된 OSS 도구들의 접근 토큰/접속 정보를 저장하고 관리하는 비밀 저장소" 역할로 설계돼 있습니다. 즉, GitLab, ArgoCD, MinIO, PostgreSQL 같은 도구를 OpenBao 경유 토큰 소스로 관리하려는 구조입니다.

1. 설치 중 연결

installing_openbao step이 활성화되면 OpenBao manifest를 클러스터에 apply합니다. 설치가 끝나면 registerStackTokenSources()가 GitLab/ArgoCD/MinIO/PostgreSQL 관련 token source 메타데이터를 DB에 등록합니다.

2. 운영 중 사용

Admin 모듈의 token source 기능이 OpenBao path를 읽고, 필요하면 회전/재발급합니다. 실제 비밀값 저장/조회 API는 shared/secrets.Router와 OpenBaoStore가 담당합니다.

설치 시점의 OpenBao 연결 알고리즘

Stack config.authentication.provider == "openbao"
  -> installing_openbao step 활성화
  -> openBaoManifest() 로 Deployment + Service YAML 생성
  -> kubectl/apply 로 클러스터에 배포
  -> runOpenBaoHealthGate() 로 라우터 health check (non-blocking)
  -> 설치 완료 후 registerStackTokenSources()

registerStackTokenSources()
  1. stack.Config 에서 어떤 OSS를 쓰는지 읽음
  2. module/provider/path 조합으로 token source 입력 생성
  3. 예: artifacts/gitlab, pipeline/argocd, artifacts/minio
  4. path 예시:
     kv/nullus/{env}/{orgID}/{module}/{provider}/token
  5. bootstrap access 정보(PostgreSQL, MinIO, ArgoCD, GitLab)도 함께 생성 가능
  6. tokenRegistry.Upsert() 로 DB에 저장

어떤 OSS 정보가 OpenBao와 연결되나

reissue 성격 token source

  • Artifacts source repository
  • Container registry
  • CI platform
  • CD tool

bootstrap access 정보

  • PostgreSQL 접속 문자열
  • MinIO endpoint / access key / secret key
  • ArgoCD admin 접근 정보
  • GitLab root 접근 정보
예시 path 규칙

kv/nullus/dev/{orgID}/pipeline/argocd/token
kv/nullus/dev/{orgID}/artifacts/minio/access

운영 시점의 OpenBao 사용 알고리즘

TokenSourceUseCase.RevealMeta(tokenSourceID)
  1. DB에서 token source 메타데이터 조회
  2. metadata.secret_manager 확인 (기본 openbao)
  3. secretRouter.GetToken(provider, path)
  4. OpenBaoStore.GetToken() 이 /v1/{mount}/data/{path} 호출
  5. token 값을 응답에 포함

TokenRotationScheduler.runIteration()
  1. DB에서 secret_manager=openbao 인 token_sources 조회
  2. next_check_at 이 지난 항목만 선택
  3. rotateOne()
  4. secret.GetToken() 으로 현재 토큰 읽기
  5. reissuer 또는 randomToken() 으로 새 토큰 생성
  6. secret.PutToken() 으로 OpenBao에 저장
  7. DB status / last_rotated_at / next_check_at 갱신
  8. token_rotation_events 기록

토큰 자동 갱신 로직

자동 갱신의 중심은 internal/admin/scheduler/token_rotation.go의 TokenRotationScheduler입니다. 역할은 "OpenBao에 저장된 token source를 주기적으로 검사하고, 교체 가능한 것은 자동으로 새 토큰으로 바꾸는 것"입니다.

TokenRotationScheduler.Start(ctx)
  1. 시작하자마자 runIteration() 1회 실행
  2. 이후 interval 마다 반복 실행

runIteration()
  1. pool / secret router / openbao provider 존재 여부 확인
  2. inFlight 플래그로 중복 실행 방지
  3. DB에서 다음 조건의 token_sources 조회
     - deleted_at IS NULL
     - metadata.secret_manager = openbao (기본값 포함)
     - next_check_at <= now()
  4. 각 항목에 대해:
     - expires_at 지났으면 expired 처리
     - manual 타입이거나 approval 필요면 failed_manual 처리
     - 아니면 rotateOne()
rotateOne(item)
  1. secret.GetToken(ctx, "openbao", item.Path) 로 현재 토큰 읽기
  2. issueToken() 으로 새 토큰 발급
  3. secret.PutToken(ctx, "openbao", item.Path, token) 로 OpenBao에 저장
  4. DB 상태 업데이트
     - 기본: rotated
     - manual 타입이면 renew_due
  5. last_rotated_at, next_check_at 갱신
  6. token_rotation_events 에 success 이벤트 기록

새 토큰 생성 방식

type == reissue 이고 provider별 재발급기가 있으면 reissuer.Reissue()를 먼저 시도합니다. 안 되면 randomToken()으로 랜덤 토큰을 만듭니다.

실패 시 재시도

실패하면 retry backoff를 기록합니다. 지연은 1분 → 5분 → 15분 → 1시간 순으로 커지고, 상태는 failed_manual로 바뀝니다.

수동 승인 흐름

manual 타입이거나 승인 필요 상태면 자동 회전을 멈추고 failed_manual 이벤트를 남깁니다. 이후 Admin API의 approve/resume 흐름으로 이어집니다.

현재 구현 상태에서 중요한 점 자동 갱신 로직 자체는 구현돼 있지만, 현재 cmd/api/main.go에는 TokenRotationScheduler를 생성해서 Start()하는 코드가 보이지 않습니다. 즉, 자동 갱신 엔진은 코드상 존재하지만 실제 서버에서 항상 켜져 있다고 단정하면 안 됩니다.

OpenBaoStore가 실제로 하는 일

Check()

/v1/sys/health와 /v1/auth/token/lookup-self를 호출해 서버 상태와 현재 토큰 유효성을 확인합니다.

PutToken()

/v1/{mount}/data/{subpath}에 {"data":{"token":...}} 형식으로 저장합니다.

GetToken()

같은 경로를 읽고 KV data 안의 token 필드를 꺼냅니다. 경로의 kv/...는 내부적으로 secret/... mount로 변환됩니다.

현재 구현 상태에서 주의할 점 관련 구조체와 함수는 많이 구현돼 있지만, 현재 cmd/api/main.go에는 OpenBao용 secrets.Router 생성/주입과 TokenRotationScheduler 시작 코드가 아직 보이지 않습니다. 즉, 설계와 핵심 로직은 존재하지만 런타임 wiring은 아직 일부 미완성일 가능성이 있습니다.

  3. echo context 의 current_user 에 저장

운영 모드 JWT 흐름

JWTAuthMiddleware()
  1. Authorization: Bearer ... 확인
  2. jwt.ParseWithClaims()
  3. kid 로 JWKS 조회
  4. issuer / audience / RS256 검증
  5. claims 에서 sub, email, preferred_username 추출
  6. provider.ExtractRoles() 로 provider별 role 해석
  7. current_user 로 context 저장

getJWKS()
  1. issuerURL 기준 캐시 확인
  2. 없으면 /protocol/openid-connect/certs 호출
  3. RSA public key 집합으로 변환
  4. 1시간 캐시

PM 관점에서 중요한 점

역할 추출은 provider 별로 다름

Keycloak과 Authentik은 JWT claims 구조가 달라서 OIDCProvider.ExtractRoles()로 추상화합니다.

RBAC는 path prefix 기반

RBACMiddleware()는 역할별로 접근 가능한 API prefix를 검사합니다.

개발/운영 간 차이

개발 모드는 헤더만 있으면 통과하지만, 운영 모드는 실제 OIDC issuer와 audience 검증이 필요합니다.

Airgap 운영 챕터

앞에서 설명한 Stack 설치 로직 중 에어갭 분기는 내부 OCI registry를 사용하는 기술 설명이었습니다. 이 챕터는 그보다 한 단계 위에서, 실제 에어갭 운영 흐름을 문서와 스크립트 기준으로 정리한 부분입니다.

무엇을 준비해야 하나

이미지 번들

필요 이미지 목록 생성, pull, save, load 절차가 필요합니다. 관련 스크립트는 airgap/scripts/00-generate-images.sh, 01-pull-images.sh, 02-save-bundle.sh, 03-load-bundle.sh 입니다.

내부 registry

10-setup-registry.sh, 12-push-to-registry.sh, 28-push-charts-oci.sh가 내부 registry와 chart/image push를 담당합니다.

클러스터와 플랫폼

11-create-cluster.sh, 21-install-nullus.sh, 22-install-platform-stack.sh, 27-install-stacks.sh로 실제 배포가 이어집니다.

에어갭 전체 흐름

1. 필요 이미지/차트 목록 산출
2. 인터넷 가능한 환경에서 이미지/차트 수집
3. 번들 생성 및 반입
4. 내부 registry 구성
5. chart OCI push
6. 이미지 registry push 또는 cluster load
7. Nullus API / DB / 플랫폼 스택 설치
8. Stack install 실행
9. Gateway / hosts / verify / port-forward 후속 작업

실제 스크립트 체인

이미지 경로
00-generate-images.sh
  -> 설치에 필요한 이미지 목록 생성

01-pull-images.sh
  -> 인터넷 가능한 환경에서 외부 registry로부터 이미지 pull

02-save-bundle.sh
  -> pull한 이미지를 tar bundle로 저장

03-load-bundle.sh
  -> 에어갭 환경으로 옮긴 bundle을 로컬 Docker/Containerd에 load

10-setup-registry.sh
  -> 내부 registry 준비

12-push-to-registry.sh
  -> 로컬에 load된 이미지를 내부 registry로 push

차트 경로
20-bundle-charts.sh
  -> chart bundle 준비

28-push-charts-oci.sh
  -> chart를 내부 OCI registry로 push

이미지 기준으로 보면

외부에서 pull -> bundle로 저장 -> 에어갭 환경에서 load -> 내부 registry로 push 순서입니다. 즉 내부 registry가 인터넷에서 직접 받는 게 아니라, 중간 번들 단계를 거칩니다.

차트 기준으로 보면

차트도 비슷하게 내부 OCI registry로 옮깁니다. 이후 Nullus 설치 시 NULLUS_HELM_OCI_REGISTRY가 이 내부 chart registry를 바라보게 됩니다.

기능 완성 판단 기준

에어갭은 단순히 설치 명령이 성공하는지보다, chart와 이미지가 모두 내부 자산으로 관리되고 외부 네트워크 없이 재현 가능해야 완성으로 볼 수 있습니다.

가장 흔한 리스크

chart는 내부 registry에 있지만 이미지 미러링이 빠져 실제 Pod가 기동하지 않는 경우, 또는 환경변수/hosts/gateway 후속 설정이 누락되는 경우입니다.

읽는 순서 추천 에어갭만 따로 이해하려면 airgap/README.md → airgap/INSTALL.md → airgap/docs/runbook.md 순서로 읽고, 그 다음 이 문서의 Stack Airgap 설치 설명을 보면 전체 그림이 가장 잘 연결됩니다.

추천 학습 순서

  1. cmd/api/main.go를 읽고 어떤 모듈이 어떤 구현체로 조립되는지 본다.
  2. Stack 설치가 가장 복잡하므로 deploy_handler.go → install_stack.go → orchestrator.go → installer.go 순서로 읽는다.
  3. CI/CD는 pipeline_handler.go → deploy_pipeline.go → applier.go를 읽는다.
  4. Observability는 dashboard_handler.go와 alert_handler.go를 먼저 보고, 그다음 repository를 읽는다.
  5. Admin은 cluster_usecase.go, token_source_usecase.go를 보면 운영 기능의 핵심이 보인다.
  6. Auth는 마지막에 auth_middleware.go와 jwt_middleware.go를 읽으면 실제 접근 제어 모델이 보인다.
PM에게 추천하는 실제 사용법 새로운 기능 논의가 생기면 먼저 이 페이지에서 해당 모듈의 진입점과 핵심 usecase를 찾고, 그 다음 실제 파일로 내려가면 됩니다. 그러면 "이 기능을 넣으면 어느 단계가 바뀌는지"를 더 빨리 파악할 수 있습니다.
