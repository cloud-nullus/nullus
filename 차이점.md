# Nullus 아키텍처-구현 차이점 분석

## 1) 분석 목적 / 전제

- 기준 문서: `Nullus 상세 기능 명세 및 시스템 아키텍처.md` (초기 아키텍처 가이드)
- 비교 대상 구현:
  - 현재 `phase1` 코드베이스
  - 최근 기능 보강 커밋 이력 (`feat/stack/devsecops-implementation`)
    - `ae3e074`, `fe8626f`, `bc51c26`, `482cc5c`, `b6c03ed`, `ab47c61`, `3157e69` 등
- 사용자 코멘트 반영: 초기 문서의 **골격은 매우 탄탄**하며, 이후 바이브 코딩 과정에서 일부 싱크가 어긋났지만 최근 패치로 기능적 공백을 상당 부분 메움.

---

## 2) 한 줄 결론

초기 문서의 큰 구조(모듈러 모놀리스, 3-Phase 설치, 버전 호환성, 리소스 산정)는 현재 구현에 대체로 살아있고,
최근 패치로 `Resource Planning + Storage + YAML View + Deploy Script + Dry Run + Gateway API/TLS + 버전 고정/보존`이 크게 강화되었습니다.
다만 **API 경로 체계, 일부 도메인/네임스페이스 모델 표현, 운영 스크립트 정합성(마이그레이션 중복 경고)**은 문서와 구현 간 불일치가 남아 있습니다.

| 영역 | 초기 명세(의도) | 현재 구현 | 차이 평가 |
|---|---|---|---|
| 시스템 골격 | UI → API(Control Plane) → Target K8s 3계층 | 동일 골격 유지 | ✅ 정합 |
| 아키텍처 스타일 | DDD 기반 모듈러 모놀리스 (`internal/{admin,stack,cicd,observability,auth}`) | 동일 구조로 구현됨 (`cmd/api/main.go` import/조립 기준) | ✅ 정합 |
| 설치 엔진 | 상태 머신 + 3-Phase(A/B/C) 프로비저닝 | `internal/stack/usecase/install_stack.go`에서 Phase A/B/C step orchestration 구현 | ✅ 정합(현실화) |
| 리소스 산정 | F10: 인원/커밋/빌드 빈도 기반 예상량 계산 | 백엔드 산정식 + 프론트 플래닝 UX 강화 (`calculate_resources.go`, install page) | ✅ 명세 대비 강화 |
| 스토리지 플래닝 | 설정 워크플로우 내 저장소 고려 | Storage 탭/연결정보/통합생성 UX 반영 | ✅ 명세 대비 구체화 |
| 설치 파일 가시성 | 설치 직전 검토 가능성 필요(암묵적) | YAML View(실제 values.yaml / manifest), 검증/역반영까지 제공 | ✅ 크게 강화 |

| 사전 검증 | 배포 전 유효성 확인 필요 | Dry Run 탭(PASS/WARN/FAIL + 최종 K8s 오브젝트) | ✅ 크게 강화 |
| 게이트웨이 | 명세는 cert-manager/TLS 맥락 중심 | Ingress 대신 Gateway API(`Gateway` + `HTTPRoute`) + TLS 체크박스 + ReferenceGrant | ✅ 현대화(명세 확장) |
| OSS 버전 관리 | Compatibility Matrix 기반 app/helm 버전 검증 | 버전 카탈로그 고정, chart/app 분리 반영, 템플릿 버전 보존, DB migration 반영 | ✅ 명세 의도에 더 근접 |
| API 경로 체계 | `/api/v1/orgs`, `/clusters`, `/templates` 등 기능별 톱레벨 | 실제는 `/api/v1/admin`, `/api/v1/stacks`, `/api/v1/cicd` 중심으로 그룹화 | ⚠️ 구조는 유사하나 경로 체계 불일치 |
| 네임스페이스 모델 | 문서에 `nullus-pipeline` 등 예시 중심 | 코드/화면은 `nullus`, `nullus-stack` 등 실행 컨텍스트별 표현 혼재 | ⚠️ 문서/구현 용어 싱크 필요 |

## 4) 최근 패치로 메워진 핵심 공백

아래는 “초기 가이드 이후 싱크가 깨졌던 부분” 중 최근에 실제로 보강된 항목입니다.

1. **리소스 산정 정교화**
   - OSS별 기본 리소스 + 수식 기반 조정 + 총합/단위 처리 강화
   - 근거: `internal/stack/usecase/calculate_resources.go`, `internal/stack/usecase/manage_resource_defaults.go`, `web/src/features/stack/pages/stack-install-page.tsx`

2. **스토리지 계획/입력 흐름 보강**
   - 기존 연결 vs 통합 생성 모드, DB/Object Storage 입력/검증 강화
   - 근거: `d26d7fd`, install page storage 관련 상태/검증 로직

3. **YAML 중심 검토/편집 UX 도입**
   - OSS별 실제 설치 파일 형태로 검토 가능
   - 근거: `ae3e074`, `fe8626f`

4. **배포 스크립트 동적 생성(EOF)**
   - 선택 옵션과 YAML 내용을 배포 스크립트에 반영
   - 근거: `bc51c26`

5. **Dry Run 최종 검토 단계 추가**
   - 체크리스트 + 최종 객체 확인으로 사전 리스크 감소
   - 근거: `482cc5c`

6. **Gateway API/TLS 적용으로 네트워크 표준화 강화**
   - Gateway/HTTPRoute 자동 생성
   - Access Domain TLS 옵션 + wildcard cert 스크립트 + ReferenceGrant 고려
   - 근거: `b6c03ed`, `ab47c61`

7. **버전 고정/보존 체계화**
   - GitLab/Argo CD 포함 app/chart 버전 분리 고정
   - 템플릿 편집 시 version 유실 방지
   - DB 마이그레이션으로 템플릿/호환성 데이터 정렬
   - 근거: `3157e69`, `db/migrations/000024_align_template_and_compatibility_versions.*`

## 5) 아직 남은 싱크 이슈 (우선순위)

### P1. API 계약 문서-코드 정합화
- 현상: 초기 명세의 endpoint taxonomy와 실제 라우팅 그룹이 다름
  - 명세: `/api/v1/orgs`, `/api/v1/templates` ...
  - 구현: `/api/v1/admin`, `/api/v1/stacks`, `/api/v1/cicd` ...
- 제안: 코드 변경보다 **문서를 구현 기준으로 업데이트**하고, 필요 시 alias/호환 엔드포인트 전략 명시

### P1. 네임스페이스/용어 정렬
- 현상: 문서 예시 네임스페이스와 실제 워크플로우 표기가 일부 다름
- 제안: “논리 네임스페이스 모델(문서)” vs “실행 기본값(구현)”을 구분 표로 명시

### P2. 운영 스크립트 정합성
- 현상: 로컬 기동 시 마이그레이션 duplicate 경고 발생 가능 (`000022_*` 파일 세트)
- 제안: 마이그레이션 파일 네이밍/중복 정책 정리 및 runbook 사전 검증 단계 강화

### P2. YAML/Deploy/DryRun 결과의 계약 스펙화
- 현상: 기능은 강하지만 산출물 스키마 계약이 문서화 부족
- 제안: `YAML View 산출 스키마`, `Dry Run 체크 규칙`, `Deploy Script 생성 규칙`을 별도 ADR/Spec으로 고정

## 6) 종합 평가

- **좋았던 점**: 초기 아키텍처 문서의 뼈대(모듈 경계, 설치 엔진, 호환성/리소스 축)가 실제 구현 방향과 잘 맞았고,
  최근 패치가 제품 실사용성(검토 가능성/배포 안전성)을 크게 끌어올림.
- **보완 포인트**: “문서 기준 API/모델 표기”와 “현재 코드 동작” 사이의 용어/계약 동기화가 다음 단계 핵심.

즉, 현재 상태는 **아키텍처 붕괴가 아니라 문서-구현 동기화 갭**에 가깝고,
기능면에서는 오히려 초기 명세보다 한 단계 진화한 부분이 많습니다.
